Most plugin pitches estimate context cost. We measured it. Every CLI command was run against 5 real projects ranging from 1 file (empty) to 17,680 files (a Next.js codebase with a 20 KB CLAUDE.md), then token-counted with tiktoken — a BPE tokenizer within ~5–10% of Claude's own.

The headline finding: the engine output is bounded. Project size, file count, and CLAUDE.md length have no meaningful effect on what enters Claude's context. A 17,680-file project costs the same as a 30-file folder.

1 · Always-on cost

Claude Code loads a skill in three layers. Only the first one is paid for on every session — even if the skill never activates.

The 27,900-token Python engine never enters context. It runs as a Bash subprocess; Claude reads only its stdout (typically 10–1,200 tokens per call, as the table below shows). The Lite tier embeds the catalog inline, trading a larger one-time SKILL.md load for zero subprocess dependency.

3 · Project size does not scale token cost

Three of the most context-sensitive commands, plotted across every project. If project size mattered, the bars would grow left-to-right. They don't.

--auto stays under 260 tokens regardless of file count. --recommend is bounded by the number of matches it finds in the catalog, not the project size. --customize is bounded by the source SKILL.md's section count, not the host project.

5 · Realistic session profiles

Using the Large project as a realistic codebase, here's what three typical session types cost.

  • Silent session (249 tokens): just project work, no skill questions. Only the frontmatter description is loaded for both tiers — the SKILL.md body never activates.
  • Light session WITH Skills Curator (5,629 tokens): SKILL.md activation + 1× --auto + 1× --recommend + 1× /skill-evaluate.
  • Heavy session WITH Skills Curator (8,382 tokens): all of the above + 2 more evaluations + 1× --audit + 1× --check + 1× --customize.
  • Heavy session WITHOUT (28,420 tokens): seven ad-hoc skill questions, each WebFetching a SKILL.md and reasoning from scratch (~4,060 tokens/question — see Methodology). No persistence → next session re-pays the same cost.
Heavy-session savings: 20,038 tokens (71%). And that's only for the current session — every evaluation also writes a registry record, so the same skill is never re-evaluated. Future sessions look up the verdict in ~150 tokens.

6 · The customization payoff

--customize is the headline capability and also where the savings story compounds. Without it, customizing a stack-mismatched skill (Vue examples in a React project) is a manual rewrite that re-reads the source SKILL.md from scratch every time.

7 · Methodology & reproducibility

Counts use tiktoken's cl100k_base encoding (GPT-4's tokenizer). Anthropic's tokenizer is also BPE with a similar vocabulary scale; measured counts are within ~5–10% of what Claude actually sees. Subprocess output is captured with capture_output=True and counted whole — this is exactly what Claude Code shows the agent.

To reproduce on your own machine:

git clone https://github.com/captkernel/Skills_Curator
cd Skills_Curator
pip install tiktoken
python docs/audit/deep_token_audit.py   # writes docs/audit/audit_results.json
python docs/audit/generate_html.py      # writes docs/token-cost-report.html

External SKILL.md size assumption (~10 KB / 2,560 tokens for the "without-skill" baseline) was validated against 42 real installed SKILL.md files on the test machine: average 11,057 bytes / 2,623 tokens, median 2,280 tokens.
