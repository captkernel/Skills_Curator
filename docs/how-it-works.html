The most important concept (and the most-missed)

There are two completely different ways code can "read" a file in a Claude Code session, and they have opposite token implications:

  1. Claude reads the file itself (via the Read, Glob, Grep tools). Every byte enters Claude's context window. A 20 KB CLAUDE.md = ~5,000 tokens consumed.
  2. A subprocess reads the file. The bytes go into the subprocess's memory (Python's RAM), get processed, and the subprocess returns only its stdout to Claude. Claude sees nothing but the stdout.

Skills Curator is built entirely around mechanism #2. When --auto runs, here's what physically happens: 

Python process starts
  ŌåÆ reads CLAUDE.md (capped at 4 KB), package.json, requirements.txt, etc. into its own RAM
  ŌåÆ parses, extracts framework tags ("nextjs", "react", "testing")
  ŌåÆ matches tags against the catalog
  ŌåÆ prints ~250 tokens of structured output to stdout
Python process exits, RAM freed
  ŌåÆ Claude sees ONLY those 250 tokens
This is the whole reason a 17,680-file project costs the same as a 30-file folder. The engine is a magnifying glass: sees a lot, reports a little. Project bytes never enter context.

Architecture

Glossary

Command (a.k.a. "verb")
A CLI flag that tells the engine what action to perform. --auto, --recommend, --customize, --check ŌĆö these are commands. In the earlier report I called them "verbs" (CLI-design jargon ŌĆö git commit: commit is a verb). Same thing; "command" is clearer.
Tier
Two implementations of the same skill ship in the plugin:
Python tier ŌĆö heavy lifting in registry.py (2,272 lines, 27,900 tokens of source). Faster on 100+ skills, regression-tested.
Lite tier ŌĆö pure-markdown SKILL.md (~11,000 tokens) that instructs the agent to do everything with Bash, Read, Glob, Grep, Write. No Python needed.
Subprocess
A separate process started by Claude via the Bash tool. Runs Python independently, has its own memory, returns only stdout/stderr/exit-code to the parent. Anything inside the subprocess is invisible to Claude's context window.
Engine vs Agent
Engine = the Python script (registry.py). Deterministic logic ŌĆö file I/O, regex matching, JSON parsing.
Agent = Claude itself. Reads SKILL.md, decides which engine command to run, interprets results, talks to the user.
Always-on cost
What loads into context at every session start, regardless of whether the skill activates. For Skills Curator, that's just the frontmatter description field of both tiers (249 tokens combined).
Activation cost
What loads when the skill is invoked. The full SKILL.md body. Pays once per session.
Catalog
The list of known skills. Combination of: 19 hand-curated entries (in source) + ~60 community skills discovered via GitHub topic search (cached 24h). Used for recommendations and search.
Trust tier
Skill source classification: ­¤Åø’ĖÅ official (Anthropic / Vercel / Microsoft / Google), Ō£ģ high (established orgs like ComposioHQ / Firecrawl), ­¤¤Ī medium, Ō¼£ community, ŌØō unknown. Used to gate recommendations ŌĆö unknown-author skills require a security scan before being surfaced.
Verdict
The output of /skill-evaluate: one of ADOPT, PARTIAL (adopt parts, skip parts), or SKIP. Persisted to ~/.claude/skills/skills-curator/registry.json so the same skill is never re-evaluated.
Fingerprint
A short hash derived from the project's key files (CLAUDE.md byte count + prefix, dependencies, etc.). --auto uses it to detect "has the project changed since last scan?" ŌĆö if not, --auto exits in ~30 tokens without doing any work.

External skills research ŌĆö how new skills are discovered

Three layers of catalog:

Layer 1: Curated catalog (built into the script)

19 hand-picked skills with pre-written pros/cons, tags, install commands, and trust tiers. Lives inside registry.py ŌĆö zero network cost, always available.

Layer 2: GitHub topic search (live, cached 24h)

Once per day, the engine hits the GitHub Search API: 

GET api.github.com/search/repositories?q=topic:claude-skill&sort=stars
GET api.github.com/search/repositories?q=topic:claude-code-skill&sort=stars
GET api.github.com/search/repositories?q=topic:agent-skill&sort=stars

Each returns ~20 repos. Engine collects just the metadata ŌĆö repo name, owner, stars, short description. SKILL.md bodies are NOT fetched here. Authors are classified into trust tiers via the _TRUSTED_AUTHORS dict:

ComposioHQ / supermemoryai / remotion-dev / firecrawl ŌåÆ Ō£ģ high

Cached at ~/.claude/skills/skills-curator/catalog.json with a 24-hour TTL. Set SKILLS_NO_TELEMETRY=1 to disable all outbound calls.

Layer 3: On-demand SKILL.md fetch

Only happens when the user runs --check, --customize, or /skill-evaluate on a specific candidate. The _read_external_skill() function tries (in order):

  1. Local path on disk
  2. Registered install path under ~/.claude/skills/
  3. owner/repo[@skill-name] ŌåÆ fetches raw.githubusercontent.com/.../SKILL.md

All reads happen in subprocess memory. The SKILL.md bytes never enter Claude's context; only the analysis result (security findings, customization plan, verdict) does.

15 regex patterns covering OWASP top-10-style risks. Files with extensions .py .sh .js .ts .md .yaml .yml .json are scanned; binary and lockfile contents are skipped. Lines marked scanner:ignore (or blocks bounded by scanner:ignore-block-start/-end) are stripped before pattern matching so the scanner doesn't flag its own pattern definitions.

--auto Proactive project scan 37-260 tokens

What it does: Fingerprints the project. Re-scans only when something changes (new dep, edited CLAUDE.md). Surfaces top 3 picks if drift detected, otherwise prints a one-line 'no changes' marker.

What it reads: CLAUDE.md, README.md, package.json, requirements.txt, pyproject.toml, Pipfile, go.mod, Cargo.toml ŌĆö each capped at 4 KB. Plus rglob counts of .py/.js/.ts/etc. for language tally.

Where reads happen: Subprocess (Python). Never in context.

What enters Claude's context: Top 3 picks if drift, or one-line 'no changes since {date}'.

Pain point it solves: User keeps re-evaluating skills every session and forgetting prior picks. --auto is cheap to run on every session; it stays silent when nothing changed.

--symptoms <phrase> Pain point ŌåÆ skill mapping 189 tokens

What it does: User describes a complaint ('tests are slow', 'ugly UI', 'deploys are manual'). The engine substring-matches against 17 hand-tuned patterns and surfaces catalog entries tagged with the matched categories.

What it reads: Just the user's phrase. No project read. No network. Pattern table is built into the script.

Where reads happen: Subprocess (Python). String matching only.

What enters Claude's context: List of catalog matches with tags (~30-190 tokens depending on match count).

Pain point it solves: User says 'X is slow' or 'I wish I had Y' instead of naming a skill. Without this, the agent has to guess what category the complaint maps to.

--recommend Project-aware ranking 29-1,140 tokens

What it does: Full project scan, ranks catalog entries by (tag-overlap ├Ś 10) + (trust-tier bonus). Splits into Capability Uplift vs Encoded Preference. Surfaces top N with pros/cons and per-section customization hints.

What it reads: Same as --auto (project signals). Plus the catalog (cached at ~/.claude/skills/skills-curator/catalog.json, 24h TTL).

Where reads happen: Subprocess (Python). Catalog read from disk cache, not network unless TTL expired.

What enters Claude's context: Ranked list with name, why-it-fits, trust tier, install command, optional customization hint.

Pain point it solves: Need a 'what skills fit here?' answer without manually scanning the catalog and reasoning about every entry.

Other questions you might have

Does the agent ever read project files itself?

Yes, sometimes. SKILL.md instructs the agent: "if the request is 'how do I do X?' and X is plausibly a skill domain, search the registry first". That's a subprocess call. But for /skill-evaluate, SKILL.md may direct the agent to read CLAUDE.md directly (using the Read tool) to assess project fit ŌĆö those bytes DO enter context. That's a deliberate trade-off: a 4ŌĆō5 KB CLAUDE.md is worth reading in full when producing a verdict, because the verdict is persistent. The subprocess can't do the judgment work; only Claude can.

Why is the Lite tier's SKILL.md so much bigger (10,988 vs 3,359 tokens)?

The Python tier delegates work to registry.py (a 27,900-token subprocess). The Lite tier has no engine ŌĆö the agent does the work using Bash, Read, Glob, Grep, Write. So the catalog, the 17 symptom patterns, the 15 security patterns, the 55-platform table, and the project-scanning instructions all have to live inline in the markdown. The trade-off: zero Python dependency, but ~3.5├Ś larger activation cost.

What if I have 1,000 installed skills?

--audit, --health, and --list output grows linearly with installed skill count. Practical guidance: at 100+ skills, prefer the Python tier (single-pass speed) over Lite (which would need many agent steps). The Python tier was regression-tested up to ~200 skills.

What stops a malicious skill from injecting prompts via its SKILL.md?

--check regex-scans for known indicators of malicious code (curl|bash, hardcoded keys, eval/exec). It doesn't currently scan for prompt-injection patterns. The trust-tier gate is the second defense ŌĆö unknown-author skills are flagged before surfacing.
